# LLM 시대의 CRM API/MCP 설계
### AI 어시스턴트를 1급 소비자로 놓는 API/MCP 전략

> 작성: 2026-06-24 · 작성자: 양시열

---

## 0. 한 장 요약

5개 고객사에 AI 어시스턴트 연동(MCP)을 베타 배포했습니다. 19일 만에 호출 1만 4천 건이 넘었고, 성공률은 99%가 넘었습니다. 잘 되는 것처럼 보였습니다.

그런데 로그를 들여다보니 문제가 보였습니다. **우리 API는 "사람이 코드로 다룬다"는 전제 위에 설계돼 있습니다.** 사람·스크립트는 데이터를 받아 코드로 처리하고 결과만 씁니다. 중간 데이터는 흘려보내도 됩니다. 그런데 채팅형 AI(claude.ai, ChatGPT 커넥터)는 받은 데이터를 코드로 처리하는 게 아니라, 받은 그대로 기억(컨텍스트)에 쌓고 직접 추론합니다. 이 작업 공간은 유한하고, 쓸데없는 데이터로 차면 터지거나 헤맵니다.

문제의 뿌리는 **서버가 결과를 "깎아서" 주지 않는다**는 데 있습니다. AI가 직접 걸러내야 하는데, 그러려면 일단 다 받아야 하고, 다 받으면 터집니다. 이 "못 깎아줌"이 세 방향으로 나타납니다:

1. **필드를 못 고른다**: 예) 레코드 조회를 하면 필요 없는 필드를 전부 내려줌
2. **행을 못 거른다**: 예) 목록 조회 API에 필터/정렬이 없기에, 원하는 이력 1건을 찾으려면 수십 건의 노이즈를 끝까지 긁어 받은 뒤 LLM이 걸러봐야함.
3. **배치로 못 묻는다**: 예) 리드 50건의 액티비티를 보려면 50번 따로 호출해야 함 (N+1)

실제로 이 반복 호출(3번)이 전체 트래픽의 69%를 차지했습니다.

HubSpot과 Salesforce는 이미 이 문제를 풀었습니다. "데이터를 통째로 넘기는" API에서 "원하는 답의 모양을 표현하면 서버가 만들어 주는" API로의 전환입니다. 우리의 답은 두 단계입니다:

- **단기**: MCP 서버가 비효율을 흡수 + 복잡한 작업은 AI가 짠 스크립트를 MCP 서버가 대신 실행해 결과만 반환
- **장기**: API를 "질문에 답하는 쿼리형"으로 재설계

## 핵심 용어 — 먼저 읽기

> 본문에 반복되는 핵심 용어입니다.

**MCP** — AI 어시스턴트와 우리 CRM API를 연결하는 표준 어댑터. *예: claude.ai에서 "세일즈맵" 커넥터를 켜면 Claude가 우리 CRM을 도구처럼 쓸 수 있습니다.*

**이 문서의 핵심 구분 — AI에는 두 종류가 있습니다:**

| | **코드 실행형** | **인-컨텍스트 도구호출형** |
|---|---|---|
| 대표 예 | **Claude Code** (개발자가 터미널·IDE에서 쓰는 코딩 에이전트) | **claude.ai · ChatGPT 커넥터** (영업담당자가 채팅창에서 "CRM 조회해줘") |
| 도구 결과 처리 | 코드로 후처리하고 **결과만 남김** | 받은 그대로 **기억(컨텍스트)에 쌓임** |
| 우리 고객 | 거의 없음 | **전부 이쪽** ← 모든 문제의 출발점 |

**컨텍스트(윈도우)** — AI가 한 번에 기억할 수 있는 유한한 작업 공간. 쓸데없는 데이터로 차면 터지거나 헤맵니다.

> 👉 이후 모든 논의는 "우리 고객 = 인-컨텍스트형"이라는 사실을 전제로 합니다.

## 1. 왜 이 문서를 쓰나 — 실제 고객이 무엇을 하려 했나

고객들이 AI에게 가장 많이 시키는 일은 단순 조회가 아닙니다. **리포트**입니다. 영업 현황을 정리하고, 담당자별 활동을 요약하고, 어떤 마케팅이 계약까지 이어졌는지 추적하는 것들입니다. 우리 CRM 안에 그 답이 있다는 걸 다들 알고 있고, 그래서 AI에게 꺼내달라고 합니다.

문제는 꺼내는 과정입니다.

**사례 ①** 한 고객사는 "지난 일주일간 영업사원이 어떤 업무를 했는지" 파악하고 싶었습니다. 담당자들이 업무를 모두 노트로 기록하기 때문에, 사실 최근 노트만 필터해서 보면 됩니다. 그런데 노트를 직접 필터하는 조회는 없습니다. 방법은 두 가지뿐입니다 — 전체 노트를 페이지네이션으로 다 끌어오거나(노트가 오래된 순서로 내려와서 최근 것을 받으려면 끝까지 다 긁어야 합니다), 또는 최근 일주일 내 수정된 리드 목록을 먼저 검색한 다음 그 리드 각각의 타임라인을 따로따로 조회하거나. 리드가 100건이면 타임라인 조회가 100번입니다. 타임라인엔 이메일 발송·수신·오픈·클릭 등 온갖 이벤트가 섞여 있어서 노트만 골라낼 수도 없습니다.

**사례 ②** 또 다른 고객사는 매일 아침 출근 전 AI가 전날 딜 현황 리포트를 만들어두게 합니다. 각 딜이 어떤 마케팅 캠페인으로 유입됐는지, 어떤 이메일 한 통이 계약을 끌고 갔는지까지를 봅니다. 실제 로그를 보면 매일 아침마다 똑같은 형태로 수백 건의 API 호출을 연속으로 날립니다. — 딜을 검색하고, 딜마다 연관 레코드를 조회하고, 레코드마다 타임라인을 조회하는 패턴이 매일 반복됩니다.

두 사례 모두 같은 구조적 문제로 귀결됩니다: **API가 "여러 레코드를 한 번에 걸러서 보여줘"를 지원하지 않아서, AI가 건별로 수백 번 돌아야 합니다.** 이 "수백 번 반복"이 1절에서 다루는 모든 문제의 실체입니다. 이 백서는 그 전모와 해법입니다.

## 2. 왜 인-컨텍스트 AI가 취약한가 — 설계 기준이 바뀐다

앞에서 두 종류의 AI를 구분했습니다. 우리 고객인 인-컨텍스트 도구호출형이 왜 취약한지가 모든 설계 결정의 출발점입니다.

사람이 데이터를 다룰 때는 "받기 → 골라내기 → 결과 보기"가 분리돼 있습니다. 중간 데이터는 흘려보내도 됩니다. 그런데 인-컨텍스트 AI는 **받는 것·골라내는 것·생각하는 것이 한 몸**입니다. 중간 데이터를 흘려보내지 못하고 전부 머릿속에 올린 채 추론해야 합니다.

> 🔧 **기술 상세** — 인-컨텍스트 LLM에게 페이지네이션이 취약한 이유: ① 각 페이지의 raw 데이터가 컨텍스트 윈도우에 누적되어 토큰을 소모합니다. ② "다음 페이지를 더 불러올지"를 모델이 매번 판단해야 하는데, 대개 첫 페이지에서 멈춥니다. ③ 컨텍스트 압축(긴 대화에서 과거 내용을 요약)이 일어나면 페이지네이션 상태 자체를 잃습니다. 즉 "완전성"을 위한 cursor 설계가 AI에겐 신뢰성 저하로 돌아옵니다.

결론: **설계는 "인-컨텍스트 AI가 받았을 때 바로 답이 되는" 모양을 기준으로 잡아야 합니다.**

## 3. 근본 문제 — "서버가 결과를 안 깎아준다"

증상은 수십 가지지만(정렬 안 됨, 건수 모름, 이력 못 찾음 등) 뿌리는 하나입니다. API가 "데이터는 다 줄 테니 골라내는 건 네가 해라"고 합니다. 사람·스크립트는 받아서 코드로 걸러내면 그만이지만, 인-컨텍스트 AI는 걸러내려면 일단 전부 컨텍스트에 올려야 합니다. 이 "못 깎아줌"이 세 방향으로 나타납니다.

### 3-1. 가로 — 필드를 못 고른다

> *"이 딜 금액이랑 상태만 알려줘"*

원본 조회는 130개가 넘는 모든 필드를 내려줍니다. 딜 20건만 조회해도 수천 개의 필드가 컨텍스트를 잠식합니다. (이 문제는 MCP가 필요한 필드만 잘라주는 방식으로 이미 상당 부분 흡수 중입니다.)

### 3-2. 세로 — 행을 못 거른다 (필터 없는 페이지네이션)

> *"이 리드 응대활동 언제 바뀌었어?"*

히스토리 조회에 필드 필터가 없습니다. "응대활동" 변경 1건을 찾으려면 담당자·시스템 자동변경 등이 섞인 전체 변경 스트림을 끝까지 긁어와 AI가 직접 골라내야 합니다. 페이지네이션(50건씩 쪼개기)은 이걸 해결해 주지 않습니다. 오히려 끝까지 긁게 만들 뿐입니다.

3-2는 3-1과 같은 병이고 원인만 다릅니다: 3-1은 열(필드)이 많아서, 3-2는 행이 많아서 컨텍스트가 터집니다.

### 3-3. 건너 — 배치로 못 묻는다 (N+1, 가장 큰 문제)

> *"최근 한 달 생성된 리드들, 담당자별로 활동 타임라인 정리해줘"*

활동 타임라인 조회는 레코드 하나만 받습니다. 그래서 리드 50건을 찾고 나면 각각 따로 50번 호출해야 합니다 — N+1 문제. 에이전트 맥락에서 이를 **Sequential Tool Dependency(순차 툴 의존)**라 부릅니다. 각 호출이 이전 결과의 ID에 의존해 직렬로 연쇄되고, 그 누적 지연을 **Agentic Latency**라 합니다. 가로·세로가 "한 결과를 못 깎음"이라면, N+1은 "여러 레코드를 한 번에 못 묻는" 같은 병의 다른 방향입니다.

실제 베타 트래픽에서 이 패턴이 지배적이었습니다.

| 지표 | 값 |
|---|---|
| 활동 타임라인 + 연관 조회 두 도구 | 전체의 **69%** |
| 21~196번 연속 호출한 묶음 | 35건 (최대 **196연속**) |
| 그 직전 도구 | 항상 "ID를 먼저 확보"하는 검색·조회 |

> 🔧 **기술 상세** — `list-engagements`·`list-associations`가 `objectId` 하나만 받는 per-record 설계라, 여러 레코드를 보려면 클라이언트가 루프를 돌 수밖에 없습니다. 텔레메트리 기준 두 도구가 5,413 + 4,622 호출로 전체의 69%, 21~196 연속 버스트가 35개였습니다. 인-컨텍스트 AI가 이 루프를 돌면 매 호출의 타임라인이 컨텍스트에 쌓여 토큰·추론이 폭증합니다.

세 방향의 공통 뿌리: **서버가 결과를 shape(필드 선택·행 필터·집계·조인)해 주지 않아, 클라이언트가 전부 받아 스스로 걸러내야 합니다.** 코드·스크립트엔 공짜지만 인-컨텍스트 AI엔 폭발입니다.

**방향은 반대여야 합니다 — 기본은 최소, 추가는 옵트인(default-minimal, opt-in expansion).** 서버가 기본적으로 최소한만 주고, 원하는 건 옵션으로 bounded하게(크기 상한 있게) 받는 방식입니다. 가로(필드)와 건너(관계)는 사실 같은 처방입니다 — 둘 다 "받고 싶은 걸 옵션으로 확장"하는 능력이 없는 것이기 때문입니다:

- 필드 확장 = **projection** (JSON:API 표준 용어로 **sparse fieldsets**. 예: `properties=[금액,상태]`) → 3-1 해결
- 관계 확장 = **expansion/include** (예: batch-read에 engagement·association을 인라인 opt-in. 관계 리소스를 별도 요청 없이 응답 본문에 포함하는 이 패턴을 JSON:API 표준에서는 **Compound Document(복합 문서)** 또는 **Resource Embedding**이라 부릅니다) → 루프 소멸 = 3-3 해결

> 🔧 **기술 상세** — HubSpot이 정확히 이 모델입니다. `properties`·associations·`propertiesWithHistory`가 전부 opt-in이고, history는 bounded(최근 20~45개). batch-read에 관계를 인라인 확장하면 필드 bloat(#1)·N+1(#3)이 한 번에 풀립니다. "expand(필드·관계)"는 Stripe의 `expand`, JSON:API의 `include`로 표준화된 패턴이며, 이 "선언적 확장"의 완성형이 5절의 governed query입니다. Anthropic은 MCP 설계 원칙으로 **"minimize round trips"**를 명시하고 있으며, sparse fieldsets + Compound Document 조합이 그 구체적 실현입니다.

5절의 "질문에 답하는 API(쿼리)"가 이 세 문제를 한 번에 풉니다 — 쿼리란 곧 "필드·행·관계를 선언하면 서버가 그 모양대로 깎아주는 것"이기 때문입니다.

## 4. 벤치마크 — 선두는 이미 풀었다

### HubSpot — REST를 AI 친화적으로 다듬음

HubSpot 최신 API엔 위 N+1이 사실상 없습니다. 네 가지를 조합해 제거했습니다:

1. **활동(노트·이메일·통화·미팅·할일)이 "1급 객체"**: 각각 독립적으로 검색 가능 → *"지난주 모든 노트"*를 루프 없이 한 번에
2. **검색 결과에 원하는 필드를 바로 포함**: "검색 후 다시 상세조회" 단계가 사라짐
3. **모든 것을 묶음(batch)으로**: 객체 100건/호출, 연관관계 1,000건/호출
4. **GraphQL**(5절 참조) — 관계를 타고 한 번에 가져오는 쿼리 인터페이스

> 🔧 **기술 상세** — Associations API v4 `POST /crm/v4/associations/{from}/{to}/batch/read`로 최대 1,000개 source ID를 한 콜에 처리합니다. 노트/이메일 등은 `/crm/v3/objects/{type}`로 search·batch-read를 지원합니다(구 v1 Engagements API는 legacy). 순(net) 패턴: 검색 1콜 → 연관 batch 1콜 → 상세 batch 1콜 = 레코드 수에 무관한 **상수 콜(O(1))**. 우리의 같은 작업은 O(N)입니다.

### Salesforce — 아예 "쿼리 언어"가 중심

Salesforce는 **SOQL**이라는 조회 전용 쿼리 언어를 플랫폼의 1급 인터페이스로 씁니다. Salesforce AI 연동(MCP)은 사실상 "쿼리 실행 도구 한 줄"에 가깝습니다.

두 회사의 공통 교훈: **"데이터를 통째로 넘기고 클라이언트가 알아서 조합"에서 "원하는 답의 모양을 표현하면 서버가 만들어 준다"로.**

## 5. 해법의 방향 — "데이터를 넘기는 API" → "질문에 답하는 API"

### 핵심 명제

> **REST 리소스 API는 "데이터를 넘기는" 모양입니다. 사람·스크립트에겐 괜찮지만, 인-컨텍스트 AI에겐 취약합니다. AI 시대의 API는 "질문에 답하는" 모양이어야 합니다.**

세 문제는 한 뿌리입니다 — API가 "루프 돌고 후처리하는 소비자"를 가정합니다. 답은 필터·정렬·집계·관계연결을 요청에 표현하면 서버가 답을 만들어 주는 질의 인터페이스, 즉 **governed query(통제된 쿼리)**입니다.

### "그냥 DB에 SQL을 때리면 되지 않나?" — 안 됩니다

가장 강력해 보이지만 위험합니다.

> 🔧 **기술 상세** — 우리 환경은 단일 DB를 모든 워크스페이스(room)가 공유하고 격리를 앱 로직으로 처리합니다. raw SQL을 노출하면: ① 모든 쿼리에 `room_id`를 신뢰된 층이 강제 주입해야 하는데 DB 레벨 안전망이 0 → AI가 한 번만 빠뜨려도 **타 고객 데이터 유출**, ② 커스텀 필드가 EAV(속성 분산 저장) 구조라 raw SQL은 그 복잡성을 그대로 노출합니다(우리 `fieldList` 추상이 가려주던 것), ③ 쓰기는 검증·계산필드·자동화(워크플로우)를 우회해 데이터를 망가뜨립니다.

정답은 **"추상화 위의 가짜 SQL"**: 우리가 정의한 스키마 위에서만 동작하고, 격리·권한·복잡도 한도를 서버가 강제하는 읽기 전용 쿼리층. HubSpot의 GraphQL, Salesforce의 SOQL이 바로 이것입니다.

> 🔧 **기술 상세** — GraphQL은 SQL이 아니라 "API용 쿼리 언어"입니다. 클라이언트가 원하는 필드·관계를 한 요청에 명시하면 서버의 resolver가 인증·격리·복잡도 가드(예: HubSpot 30,000 points)를 강제하며 내부에서 조립해 돌려줍니다. `GraphQL : API = SQL : DB` — 같은 질의 표현력을, 한 단계 위(애플리케이션 스키마)에서, 안전하게. 단 HubSpot의 GraphQL은 CMS Hub에 뿌리를 둔 곁가지이고, "쿼리를 읽기의 주 인터페이스로" 삼은 더 순수한 예는 Salesforce SOQL입니다.

### 읽기와 쓰기는 대칭이 아닙니다 (중요)

쿼리 이야기는 전부 "읽기"입니다. **쓰기(생성/수정/삭제)는 DB·쿼리로 직접 하면 절대 안 됩니다.**

| | **읽기(READ)** | **쓰기(WRITE)** |
|---|---|---|
| 적정 방식 | 쿼리(가짜 SQL / GraphQL) | 타입이 정해진 개별/묶음 API |
| 데이터에 가까이? | 가도 됨 | 안 됨 — 반드시 로직층 통과 |

쓰기는 검증·계산필드·자동화(워크플로우/이메일 발송)·권한·변경 이력 생성을 포함합니다. DB로 직접 쓰면 이 모두를 우회해 데이터가 깨지고 자동화가 망가집니다 — 아이러니하게도 우리가 그토록 보여주려던 "변경 이력"이 안 남습니다. HubSpot·Salesforce 모두 쿼리는 읽기 전용, 쓰기는 타입드 API로 분리합니다.

## 6. 지금의 타협점 — AI가 짠 스크립트를 우리가 대신 실행

쿼리형 API는 백엔드 재설계가 필요해 시간이 걸립니다. 그 전까지의 현실적인 다리가 있습니다. 업계에선 이를 **"코드 실행(Code Execution with MCP)"**이라 부릅니다 (Anthropic이 2025년 정식화한 패턴).

아이디어는 간단합니다. 복잡한 작업은 AI가 도구를 일일이 호출하는 대신, API 사용법을 보고 스크립트(코드)를 짜고, 그 스크립트를 실행 환경이 돌려 결과만 AI에게 돌려줍니다. 반복 루프가 코드 안에서 일어나니 raw 데이터가 AI의 작업 공간을 더럽히지 않습니다.

실행 환경은 두 가지입니다:
- **(i)** AI 쪽(Claude Code 같은 코드 실행형) — 단 우리 고객은 채팅형이라 대부분 불가합니다.
- **(ii) 우리 MCP 서버가 직접 실행** — 우리 고객에게 유일하게 먹히는 형태입니다. 우리 서버 = 그 "for문을 돌리는 환경"입니다.

### 안전 설계 — "AI가 짠 코드를 우리 서버가 실행" = 본질적으로 위험

잘못하면 서버 장악(코드 주입) 사고입니다. 그래서 이 도구(`salesmap-run-script`)는 네 가지를 반드시 지켜야 합니다:

1. **격리 실행** — 단순 eval 금지. 인터넷 접근은 우리 API로만 잠금.
2. **정해진 SDK만 허용** — AI 코드는 우리가 감싼 함수(`sm.search()` 등)만 호출. 임의 인터넷·DB 접근 불가.
3. **읽기 전용** — 스크립트엔 쓰기 함수를 아예 주지 않음 (7절).
4. **시간·자원 한도 + 에러 반환** — 무한 루프 차단, 실패 시 AI가 고칠 수 있게 에러 반환.

> 🔧 **기술 상세** — `run-script(script)` 도구. script는 격리 런타임(isolated-vm/QuickJS 또는 E2B·Vercel Sandbox 같은 샌드박스)에서 실행, egress는 salesmap.kr로 lock. 노출 SDK는 client.ts의 read 함수만 래핑(rate-limit·retry·auth 재사용). raw `fetch` 차단 → 스크립트가 write를 칠 통로 자체가 없습니다. Salesforce의 "SQL 한 줄 도구"의 사촌이되, 표현력↑·안전성↓이라 임시(bridge) 용도이고, governed query가 종착입니다.

### 이건 무엇을 풀고 무엇을 못 푸나

- ✅ AI의 작업 공간(컨텍스트) 오염을 막습니다. 결과만 돌아옵니다.
- ❌ 네트워크 반복(N+1) 자체는 그대로입니다. 196번 호출은 서버 안에서 여전히 일어나 비용·지연·한도에 걸립니다.

그래서 bridge이고, 진짜 해결은 쿼리형 API입니다.

## 7. 어디서 풀까 — MCP 서버 vs 백엔드 API

| 문제 | MCP 레이어로? | 백엔드/쿼리층 필요? |
|---|---|---|
| 응답 비대화 (전 필드) | ✅ 필요한 필드만 잘라줌 (이미 함) | — |
| 페이지·건수·정렬 | △ 서버가 흡수 가능하나 호출·지연 비용 존재 | ✅ 쿼리가 깔끔 |
| per-record N+1 | △ 서버 루프로 컨텍스트만 보호, 네트워크 비용 그대로 | ✅ 묶음/쿼리만이 제거 |
| 종착 | — | ✅ 통제된 쿼리 API |

> 한 줄: **MCP는 "응답의 모양"은 고치지만 "데이터 접근 패턴"은 못 고칩니다.** 후자는 API가 리소스가 아니라 질문을 받게 바뀌어야 풀립니다.

## 8. 결론

우리가 MCP로 발견한 건 "AI가 우리 API를 잘 못 쓴다"가 아니라 **"우리 API가 사람·스크립트 시대의 가정 위에 서 있다"**는 사실입니다. 인-컨텍스트 채팅형 AI는 루프를 돌고 데이터를 후처리하지 못합니다.

그래서 차기 API는 인-컨텍스트 AI를 1급 소비자로 놓고 설계해야 합니다:
- 데이터를 통째로 넘기지 말고 질문에 답하라 (통제된 쿼리).
- 각 도구는 한 번의 호출로 AI에게 필요한 답을 완결시켜야 합니다 — **Self-contained Tool(자립형 도구)** 원칙이자 "minimize round trips"의 실천입니다. Sequential Tool Dependency를 툴 설계 단계에서 차단합니다.
- MCP가 응답 모양을 다듬고, 복잡한 작업은 AI의 스크립트를 서버가 대신 실행합니다.
