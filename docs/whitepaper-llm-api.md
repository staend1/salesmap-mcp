### AI 어시스턴트를 1급 소비자로 놓는 API/MCP 전략

> 작성: 2026-06-24 · 작성자: 양시열

## 0. 한 장 요약

5개 고객사에 AI 어시스턴트 연동(MCP)을 베타 배포했습니다. 19일 만에 호출 1만 4천 건이 넘었고, 성공률은 99%가 넘었습니다. 잘 되는 것처럼 보였습니다.

그런데 로그를 들여다보니 문제가 보였습니다. **우리 API는 "사람이 코드로 다룬다"는 전제 위에 설계돼 있습니다.** 사람·스크립트는 데이터를 받아 코드로 처리하고 결과만 씁니다. 중간 데이터는 흘려보내도 됩니다. 그런데 채팅형 AI(claude.ai, ChatGPT 커넥터)는 받은 데이터를 코드로 처리하는 게 아니라, 받은 그대로 기억(컨텍스트)에 쌓고 직접 추론합니다. 이 작업 공간은 유한하고, 쓸데없는 데이터로 차면 터지거나 헤맵니다.

문제의 뿌리는 **서버가 결과를 "깎아서" 주지 않는다**는 데 있습니다. "못 깎아줌"이 세 방향으로 나타납니다:

1. **필드를 못 고른다**: 레코드 조회를 하면 필요 없는 필드를 전부 내려줌
2. **행을 못 거른다**: 필터·정렬 없이 전체를 긁어야 원하는 1건을 찾을 수 있음
3. **배치로 못 묻는다**: 리드 50건의 활동을 보려면 50번 따로 호출 (N+1)

실제 트래픽에서 이 반복 호출(3번)이 전체의 69%를 차지했습니다.

해법은 두 단계입니다:

- **차선(지금 당장)**: 기존 REST API에 `fields`, `include` 옵션을 붙이고 MCP로 래핑 — AI가 "원하는 깊이"를 한 번의 호출로 선언해서 받아감
- **최선(종착)**: API를 "질문에 답하는 쿼리형"으로 재설계 (governed query)

## 핵심 용어 — 먼저 읽기

> 본문에 반복되는 핵심 용어입니다.

**MCP** — AI 어시스턴트와 우리 CRM API를 연결하는 표준 어댑터. *예: claude.ai에서 "세일즈맵" 커넥터를 켜면 Claude가 우리 CRM을 도구처럼 쓸 수 있습니다.*

**이 문서의 핵심 구분 — AI에는 두 종류가 있습니다:**

| | **코드 실행형** | **인-컨텍스트 도구호출형** |
|---|---|---|
| 대표 예 | **Claude Code** (개발자가 터미널·IDE에서 쓰는 코딩 에이전트) | **claude.ai · ChatGPT 커넥터** (영업담당자가 채팅창에서 "CRM 조회해줘") |
| 도구 결과 처리 | 코드로 후처리하고 **결과만 남김** | 받은 그대로 **기억(컨텍스트)에 쌓임** |
| 우리 고객 | 거의 없음 | **전부 이쪽** ← 모든 문제의 출발점 |

**컨텍스트(윈도우)** — AI가 한 번에 기억할 수 있는 유한한 작업 공간. 쓸데없는 데이터로 차면 터지거나 헤맵니다.

> 👉 이후 모든 논의는 "우리 고객 = 인-컨텍스트형"이라는 사실을 전제로 합니다.

## 1. 왜 이 문서를 쓰나 — 실제 고객이 무엇을 하려 했나

고객들이 AI에게 가장 많이 시키는 일은 단순 조회가 아닙니다. **리포트**입니다. 영업 현황을 정리하고, 담당자별 활동을 요약하고, 어떤 마케팅이 계약까지 이어졌는지 추적하는 것들입니다. 우리 CRM 안에 그 답이 있다는 걸 다들 알고 있고, 그래서 AI에게 꺼내달라고 합니다.

문제는 꺼내는 과정입니다.

**사례 ①** 한 고객사는 "지난 일주일간 영업사원이 어떤 업무를 했는지" 파악하고 싶었습니다. 담당자들이 업무를 모두 노트로 기록하기 때문에, 사실 최근 노트만 필터해서 보면 됩니다. 그런데 노트를 직접 필터하는 조회는 없습니다. 방법은 두 가지뿐입니다 — 전체 노트를 페이지네이션으로 다 끌어오거나, 최근 일주일 내 수정된 리드 목록을 먼저 검색한 다음 그 리드 각각의 타임라인을 따로따로 조회하거나. 리드가 100건이면 타임라인 조회가 100번입니다.

**사례 ②** 또 다른 고객사는 매일 아침 전날 딜 현황 리포트를 AI가 만들어두게 합니다. 각 딜이 어떤 캠페인으로 유입됐는지, 어떤 이메일 한 통이 계약을 끌고 갔는지까지 봅니다. 실제 로그를 보면 매일 아침마다 수백 건의 API 호출이 연속으로 날립니다. 딜을 검색하고, 딜마다 연관 레코드를 조회하고, 레코드마다 타임라인을 조회하는 패턴이 매일 반복됩니다.

두 사례 모두 같은 구조적 문제로 귀결됩니다: **API가 "여러 레코드를 한 번에 걸러서 보여줘"를 지원하지 않아서, AI가 건별로 수백 번 돌아야 합니다.**

## 2. 왜 인-컨텍스트 AI가 취약한가 — 설계 기준이 바뀐다

사람이 데이터를 다룰 때는 "받기 → 골라내기 → 결과 보기"가 분리돼 있습니다. 중간 데이터는 흘려보내도 됩니다. 그런데 인-컨텍스트 AI는 **받는 것·골라내는 것·생각하는 것이 한 몸**입니다. 중간 데이터를 흘려보내지 못하고 전부 머릿속에 올린 채 추론해야 합니다.

결론: **설계는 "인-컨텍스트 AI가 받았을 때 바로 답이 되는" 모양을 기준으로 잡아야 합니다.**

---

## 3. 근본 문제 — "서버가 결과를 안 깎아준다"

증상은 수십 가지지만 뿌리는 하나입니다. API가 "데이터는 다 줄 테니 골라내는 건 네가 해라"고 합니다. 사람·스크립트는 받아서 코드로 걸러내면 그만이지만, 인-컨텍스트 AI는 걸러내려면 일단 전부 컨텍스트에 올려야 합니다. 이 "못 깎아줌"이 세 방향으로 나타납니다.

### 3-1. 가로 — 필드를 못 고른다

> *"이 딜 금액이랑 상태만 알려줘"*

원본 조회는 130개가 넘는 모든 필드를 내려줍니다. 딜 20건만 조회해도 수천 개의 필드가 컨텍스트를 잠식합니다.

### 3-2. 세로 — 행을 못 거른다 (필터 없는 페이지네이션)

> *"이 리드 응대활동 언제 바뀌었어?"*

히스토리 조회에 필드 필터가 없습니다. "응대활동" 변경 1건을 찾으려면 전체 변경 스트림을 끝까지 긁어와 AI가 직접 골라내야 합니다.

### 3-3. 건너 — 배치로 못 묻는다 (N+1, 가장 큰 문제)

> *"최근 한 달 생성된 리드들, 담당자별로 활동 타임라인 정리해줘"*

활동 타임라인 조회는 레코드 하나만 받습니다. 리드 50건을 찾고 나면 각각 따로 50번 호출해야 합니다 — N+1 문제. 에이전트 맥락에서 이를 **Sequential Tool Dependency(순차 툴 의존)**라 부릅니다. 각 호출이 이전 결과의 ID에 의존해 직렬로 연쇄되고, 그 누적 지연을 **Agentic Latency**라 합니다.

실제 베타 트래픽에서 이 패턴이 지배적이었습니다.

| 지표 | 값 |
|---|---|
| 활동 타임라인 + 연관 조회 두 도구 | 전체의 **69%** |
| 21~196번 연속 호출한 묶음 | 35건 (최대 **196연속**) |
| 그 직전 도구 | 항상 "ID를 먼저 확보"하는 검색·조회 |

세 방향의 공통 뿌리: **서버가 결과를 shape(필드 선택·행 필터·관계 포함)해 주지 않아, AI가 전부 받아 스스로 걸러내야 합니다.**

## 4. 벤치마크 — 선두는 이미 풀었다

### HubSpot — REST를 AI 친화적으로 다듬음

HubSpot 최신 API엔 위 N+1이 사실상 없습니다. 네 가지를 조합해 제거했습니다:

1. **활동(노트·이메일·통화·미팅·할일)이 "1급 객체"**: 각각 독립적으로 검색 가능 → *"지난주 모든 노트"*를 루프 없이 한 번에
2. **검색 결과에 원하는 필드를 바로 포함**: "검색 후 다시 상세조회" 단계가 사라짐
3. **모든 것을 묶음(batch)으로**: 객체 100건/호출, 연관관계 1,000건/호출
4. **GraphQL** — 관계를 타고 한 번에 가져오는 쿼리 인터페이스

같은 작업 기준: 우리는 O(N)번 호출, HubSpot은 **O(1)번**.

### Salesforce — 아예 "쿼리 언어"가 중심

Salesforce는 **SOQL**이라는 조회 전용 쿼리 언어를 플랫폼의 1급 인터페이스로 씁니다. Salesforce AI 연동(MCP)은 사실상 "쿼리 실행 도구 한 줄"에 가깝습니다.

두 회사의 공통 교훈: **"데이터를 통째로 넘기는" API에서 "원하는 답의 모양을 표현하면 서버가 만들어 주는" API로.**

## 5. 해법 — Resource API에서 Query API로

### 핵심 명제

> **REST 리소스 API는 "데이터를 넘기는" 모양입니다. AI 시대의 API는 "질문에 답하는" 모양이어야 합니다.**

방향은 **default-minimal, opt-in expansion** — 기본은 최소한만 주고, 원하는 건 옵션으로 bounded하게(크기 상한 있게) 추가하는 방식입니다.

### 차선 — fields + include 옵션 → MCP 래핑

현실적인 첫 단계입니다. 기존 REST tool에 두 가지 파라미터를 붙입니다:

| 레이어 | 지금 (Resource API) | 개선 후 (Query option) |
|---|---|---|
| 기본 검색 | search → id, name만 | `search(fields: [email, phone, ...])` |
| Association | get_record → 별도 get_association | `get_record(include: [deals, org])` |
| Activity | get_record → 별도 get_activity | `get_record(include: [notes, emails])` |

- **`fields`** — 반환할 필드 선택 (JSON:API 표준 용어로 **sparse fieldsets**). 3-1 해결.
- **`include`** — 관련 리소스를 응답 본문에 인라인으로 포함 (JSON:API 표준에서 **Compound Document** 또는 **Resource Embedding**). 3-3 해결.

아이디어는 "tool 수를 줄이는 게 아니라, tool의 응답 깊이를 호출자가 조율한다"입니다. AI가 "이메일도 필요해"라고 하면 지금은 search 후 각 id마다 재호출 = 2n hop이지만, include 옵션 하나로 → 1 hop.

이렇게 만든 API를 MCP tool로 래핑하면, AI가 한 번의 tool 호출로 원하는 깊이의 답을 완결해서 받아갑니다. Anthropic이 MCP 설계 원칙으로 명시한 **"minimize round trips"**의 구체적 실현입니다.

### 최선 — Governed Query (종착)

차선의 `fields + include`는 파라미터 조합이 고정돼 있습니다. 종착은 AI가 원하는 모양을 **쿼리 언어로 자유롭게 표현**하면 서버가 만들어 주는 구조입니다.

단 "그냥 DB에 SQL을 때리면 되지 않나?"는 안 됩니다. 우리 환경은 단일 DB를 모든 워크스페이스가 공유하고 격리를 앱 로직으로 처리합니다. Raw SQL 노출은 고객 데이터 격리 보장이 불가능합니다.

정답은 **"추상화 위의 가짜 SQL"**: 우리가 정의한 스키마 위에서만 동작하고, 격리·권한·복잡도 한도를 서버가 강제하는 읽기 전용 쿼리층입니다. HubSpot의 GraphQL, Salesforce의 SOQL이 이것입니다.

`fields + include`가 "선언할 수 있는 모양이 정해진" 쿼리라면, governed query는 "모양 자체를 자유롭게 선언하는" 쿼리입니다.

### 읽기와 쓰기는 대칭이 아닙니다

쿼리 이야기는 전부 "읽기"입니다. **쓰기(생성/수정/삭제)는 쿼리로 직접 하면 안 됩니다.** 쓰기는 검증·계산필드·자동화(워크플로우)·변경 이력 생성을 포함합니다. 이걸 우회하면 데이터가 깨지고 자동화가 망가집니다.

| | **읽기(READ)** | **쓰기(WRITE)** |
|---|---|---|
| 방식 | 쿼리 (fields·include·governed query) | 타입이 정해진 개별/묶음 API |
| 데이터에 가까이? | 가도 됨 | 반드시 로직층 통과 |

## 6. 결론

우리가 MCP로 발견한 건 "AI가 우리 API를 잘 못 쓴다"가 아니라 **"우리 API가 사람·스크립트 시대의 가정 위에 서 있다"**는 사실입니다.

그래서 차기 API는 인-컨텍스트 AI를 1급 소비자로 놓고 설계해야 합니다:

- **차선**: 기존 REST API에 `fields + include` 옵션 추가 → MCP로 래핑 → AI가 1 hop으로 완결
- **최선**: governed query — 원하는 모양을 선언하면 서버가 격리·권한을 보장하며 만들어 줌
- **쓰기는 항상**: 타입드 API로, 검증·자동화 로직층을 통과해서

각 도구는 한 번의 호출로 AI에게 필요한 답을 완결시켜야 합니다 — **Self-contained Tool(자립형 도구)** 원칙이자 Sequential Tool Dependency를 툴 설계 단계에서 차단하는 방법입니다.
