# 우리 API는 사람을 위해 잘 만들어졌다

**세일즈맵 MCP를 만들면서 배운 것들 — AI는 사람이 아니었다**

---

19일. 14,500건. 성공률 99%.

수치만 보면 성공이었다. 우리 CRM에 AI 어시스턴트를 연결하는 MCP 서버를 5개 고객사에 베타로 배포했고, 약 3주 만에 1만 4천 건이 넘는 API 호출이 들어왔다. 오류는 거의 없었다.

그런데 로그를 열어봤다.

---

## 고객들이 실제로 하려는 것

우리가 예상했던 사용은 이랬다. 영업담당자가 claude.ai를 열고 "이 리드 상태가 어떻게 돼 있어?"라고 묻는다. AI가 CRM을 조회해서 답을 준다. 단순하고 깔끔하다.

실제 사용은 달랐다. 고객들이 AI에게 가장 많이 시키는 건 단순 조회가 아니라 **리포트**였다.

한 고객사 담당자는 매주 월요일 아침마다 같은 요청을 했다. *"지난 일주일간 영업팀이 어떤 업무를 했는지 정리해줘."* 팀원들이 모든 업무를 CRM 노트로 기록하기 때문에, 사실 최근 노트만 모아보면 된다. CRM 안에 다 있는 데이터다.

또 다른 고객사는 아예 자동화를 원했다. *"매일 아침 출근 전에 전날 딜 현황을 정리해둬. 각 딜이 어떤 마케팅 채널로 들어왔는지, 어떤 이메일이 계약으로 이어졌는지까지."* 마케팅 성과는 보통 "리드를 몇 건 넘겼나"에서 끊긴다. 그 뒤로 어떻게 됐는지는 늘 깜깜하다. CRM엔 그 이후 흐름이 다 있는데, 꺼내서 잇기가 어려웠을 뿐이다.

우리 텔레메트리 로그에는 이 두 번째 고객사의 흔적이 선명하게 찍혀 있다. 매일 오전 8~9시, 영업 시작 전 정각에, 수백 건의 API 호출이 연속으로 쏟아진다. 영업일이면 빠짐없이. 3주 내내.

---

## 로그를 보니 뭔가 이상했다

이 고객사의 어느 날 아침 로그를 들여다보면 패턴이 보인다. 딜 목록을 검색한다. 각 딜에 연결된 레코드를 조회한다. 그 레코드 각각의 활동 타임라인을 조회한다. 딜이 50건이면 타임라인 조회도 50번이다.

전체 베타 트래픽에서 이 "활동 타임라인 조회"와 "연관 레코드 조회" 두 도구가 차지하는 비중이 **69%** 였다. 21번에서 많게는 196번까지 연속으로 호출하는 묶음이 35개나 됐다.

AI가 잘못 쓰는 거라고 생각했다. 처음엔.

그런데 AI 탓이 아니었다.

---

## 소비자가 바뀌었다

우리 API는 사람이 코드로 다룬다는 전제 위에 설계됐다. 당연한 전제였다. API는 원래 그런 거다.

사람이나 스크립트가 API를 쓸 때는 이렇게 된다. 데이터를 받는다 → 코드로 처리한다 → 결과만 남긴다. 중간에 필요 없는 데이터는 흘려보내면 그만이다. 딜 50건의 타임라인이 필요하면 루프를 돌면 된다. 결과만 추려서 보고서를 만들면 된다.

**채팅형 AI는 다르다.**

claude.ai 같은 AI 커넥터를 쓰는 영업담당자를 생각해보자. 이 AI는 코드를 실행하지 않는다. 도구에서 받은 데이터를 코드로 처리하는 게 아니라, 받은 그대로 "기억(컨텍스트)"에 쌓고 직접 추론한다. 이 작업 공간은 유한하다. 쓸데없는 데이터로 차면 터지거나 헤맨다.

그래서 채팅형 AI는 루프를 돌 수 없다. 50번 호출해서 결과를 모아도, 그 50개 응답이 전부 기억 속에 쌓인다. 스크립트는 결과만 남기고 중간 데이터를 버리지만, AI는 중간 과정을 전부 기억한 채 생각해야 한다.

우리 베타 고객 5곳은 Claude Code 같은 코드 실행형 AI를 쓰는 개발자가 아니다. 전부 채팅창에서 AI에게 말을 거는 영업담당자다. 모든 문제의 출발점이 여기에 있었다.

---

## 세 가지 병

우리 API가 이 채팅형 AI에게 불편한 이유를 들여다보면, 결국 같은 뿌리에서 세 방향으로 나타난다.

**첫째, 필드를 못 고른다.** 딜 하나를 조회하면 관련 필드 130개가 전부 내려온다. 금액이랑 상태만 알고 싶어도 130개를 다 받는다. 사람은 코드로 두 개만 골라내면 되지만, AI는 130개가 전부 기억 속으로 들어온다. 딜 20건 조회하면 수천 개의 필드가 한꺼번에 쌓인다. 우리 MCP 서버에서 자주 쓰는 필드만 잘라주는 방식으로 이미 상당 부분을 흡수했다. 이건 MCP 레이어에서 풀 수 있는 문제다.

**둘째, 행을 못 거른다.** 노트나 변경 이력 같은 건 필터 없이 전체 스트림으로 내려온다. "담당자가 언제 바뀌었어?"를 물으면 시스템 자동변경, 유입경로 수정, 팀 배정 같은 온갖 변경 이력이 섞여서 내려온다. 원하는 것 1건을 찾으려면 전부 받아봐야 한다. 50건씩 쪼개주는 페이지네이션은 이걸 해결해 주지 않는다. 오히려 AI가 50건씩 계속 받아야 하는 구조가 된다.

**셋째, 배치로 못 묻는다.** 이게 가장 크다. 활동 타임라인 조회는 레코드 하나만 받는다. 딜 50건의 타임라인을 보고 싶으면 50번 따로 호출해야 한다. API 구조 자체가 이 패턴을 강제한다. 스크립트는 루프 돌면 그만이지만, AI는 50번 호출하면 50개의 응답이 전부 기억에 쌓인다.

세 문제는 다 같은 말이다. **API가 "데이터는 다 줄 테니 골라내는 건 네가 해라"고 설계돼 있는데, AI는 골라내려면 일단 전부 받아야 한다.**

---

## 선두주자들은 이미 알고 있었다

우리가 처음 이 문제를 인식했을 때, HubSpot API를 뜯어봤다.

HubSpot 최신 API에는 위에서 말한 N+1 문제가 사실상 없다. 몇 가지를 조합해서 제거했다.

노트, 이메일, 미팅 같은 활동 기록이 독립적으로 검색 가능한 1급 객체다. "지난주 작성된 모든 노트"를 루프 없이 한 번에 가져올 수 있다. 검색 결과에 원하는 필드를 바로 포함시킬 수 있어서 "검색 후 다시 상세조회"하는 단계가 사라진다. 연관 레코드는 최대 1,000건을 한 번에 조회할 수 있다.

결과적으로 딜 목록에서 연결된 이메일까지 보는 작업이 "검색 1번 → 연관 조회 1번 → 상세 조회 1번 = 3번"으로 끝난다. 딜이 50건이든 500건이든 호출 수는 똑같다.

Salesforce는 아예 다른 방향으로 갔다. SOQL이라는 조회 전용 쿼리 언어를 플랫폼의 1급 인터페이스로 쓴다. Salesforce AI 연동을 보면 사실상 "이 쿼리를 실행해줘" 도구 하나가 전부다. 원하는 데이터의 모양을 표현하면 서버가 만들어서 준다.

두 회사가 서로 다른 방법으로 같은 방향을 가리키고 있다. **"데이터를 통째로 주는" API에서 "원하는 답의 모양을 표현하면 서버가 만들어 주는" API로.**

---

## 우리는 지금 뭘 하고 있나

솔직히 말하면, API를 당장 다시 만들 수는 없다. 그래서 단계를 나눠서 접근하고 있다.

**지금 MCP 레이어에서**: 원하는 필드를 인자로 받아, 백엔드에서 전체를 다 받은 다음 MCP 서버가 요청한 것만 잘라서 돌려주는 방식으로 필드 문제를 흡수했다. API 자체는 안 바꾸고, 레이어에서 대신 필터링하는 것이다. 페이지네이션이 있을 때는 "결과가 더 있다, 계속 조회해라"는 힌트를 응답에 명시적으로 담아서 AI가 첫 페이지에서 "없네"하고 멈추는 걸 막았다.

**백엔드에서 빠르게 고칠 수 있는 것**: 검색할 때 원하는 필드를 바로 받을 수 있게 하면 "검색 후 다시 상세조회"하는 N+1이 대부분 사라진다. 활동 타임라인에 필터를 추가하면 세로 문제도 해결된다. 이 두 개만 해도 베타 트래픽의 N+1 상당 부분이 없어질 거라고 본다.

**장기적으로**: "원하는 답의 모양을 표현하면 서버가 만들어 주는" 쿼리형 API 방향으로 가야 한다. 이게 맞는 방향이라는 건 HubSpot과 Salesforce가 이미 증명했다.

한 가지 더 있다. Anthropic이 2025년에 **"Code Execution with MCP"** 라는 패턴을 발표했다. 복잡한 작업은 AI에게 도구를 하나씩 호출하게 시키는 대신, AI가 스크립트를 짜고 MCP 서버가 그걸 대신 실행해서 결과만 돌려주는 방식이다. 반복 루프가 서버 안에서 돌기 때문에 중간 데이터가 AI 기억에 쌓이지 않는다. 매일 아침 딜 리포트처럼 "여러 레코드를 순회해서 취합"하는 작업에 딱 맞는 패턴이다. 물론 이 방식도 API 호출 자체가 N번 일어나는 건 막을 수 없다. 컨텍스트 오염은 막지만, 근본 해결은 아니다. 그래서 쿼리형 API로 가는 길 위의 현실적인 다리로 보고 있다.

## 결론

이 글을 쓰게 된 계기는 베타 로그였다. 잘 되는 줄 알았는데 들여다보니 전체 호출의 69%가 같은 레코드를 수십~수백 번 반복 조회하는 패턴이었다.

AI가 우리 API를 잘 못 쓰는 게 아니었다. **API가 AI를 소비자로 생각한 적이 없었던 것이다.**

우리 API는 잘 만들어졌다. 사람과 스크립트를 위해서. 그 전제가 오래됐을 뿐이다. 채팅창에서 "이번 달 딜 현황 정리해줘"라고 말하는 영업담당자가 소비자가 된 순간부터, 게임의 룰이 바뀌었다.

다음에 API를 새로 설계할 때는 한 가지를 먼저 물어볼 것 같다. *"이걸 채팅 AI가 받았을 때 바로 답이 되는가?"*
